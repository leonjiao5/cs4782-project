--- dora_layers ---
core equation (Eq. 5 in paper):
    W' = m * (W0 + s*BA) / ||W0 + s*BA||_c
  where:
    - W0 ∈ R^(out, in) is the frozen pretrained weight
    - B ∈ R^(out, r), A ∈ R^(r, in)  — low-rank LoRA factors
    - s = alpha / rank  (LoRA scaling; paper folds it into BA, but we keep it explicit)
    - m ∈ R^(out,)     — trainable per-output-feature magnitude
    - ||.||_c          — column-wise L2 norm. NVlabs ref uses weight.norm(p=2, dim=1)
                         on PyTorch (out, in) weight, giving one scalar per output feature.
                         existing dora_layers.py init matches this convention -- keep.

init (must give W' == W0 at step 0):
- m = ||W0||_c       (frozen-weight column norms; existing code is correct)
- A ~ kaiming_uniform_(a=0)
- B = 0              (so s*BA = 0, numerator = W0, denom = ||W0||_c, m cancels norm -> W0)

trainable: m, lora_A, lora_B. frozen: base.weight, base.bias.
per-layer trainable count: out + r*(in + out)

detach trick (Eq. 11, sec 4.3) -- REQUIRED:
- treat ||V + dV||_c = ||W0 + s*BA||_c as a CONSTANT (call .detach()) in the denominator
- ablation in paper: -24.4% memory on LLaMA-7B, ~0.2pt accuracy delta
- without this, gradient of ||.||_c flows through B and A and blows up training memory

forward (efficient, single matmul):
    weight_eff = W0 + s*BA
    c = weight_eff.norm(p=2, dim=1).detach()      # shape (out,)
    weight_scaled = (m / c).unsqueeze(1) * weight_eff   # broadcast over rows
    y = F.linear(self.dropout(x), weight_scaled, base.bias)
- input x can be (B, T, in); F.linear handles batched shapes
- dropout applied to LoRA branch input, matching peft DoRA convention
- dropout p = 0.05 default (paper Table 8)

merge / unmerge (zero inference overhead):
- merge(): bake m * weight_eff / c into base.weight, set self.merged = True
- unmerge(): restore W0 (cache it on init or recompute from m,A,B)
- inference path uses merged weights -> identical latency to base model

gradient sanity check (do this BEFORE scaling up):
- run scratch DoRA vs peft.LoraConfig(use_dora=True) on a 2-layer toy
- same init, same input -> grads on m/A/B should match within fp tolerance

- freeze everything except DoRA params (m, A, B). do NOT train lm_head.
- expected trainable %: ~0.71% for LLaMA3-8B at r=16 / target=QKVUD (paper Table 1)

--- train ---
optimizer / schedule:
- AdamW, cosine schedule, ~3% warmup
  (paper uses Linear + 100 warmup steps; we keep cosine -- more standard for math fine-tuning)
- precision: bf16 on A100/H100, fp16 + 4-bit base on T4
- gradient checkpointing on
- save every N steps; eval AIME every M steps

hyperparameters (paper Table 8 anchors; tune for AIME):
- rank r: 16 default. always alpha = 2r in the paper.
- alpha: 32 (paired with r=16)
- dropout: 0.05
- batch size: 16 effective (use grad accum if device-batch < 16)
- epochs: 3 (adjust if our train corpus is materially larger than paper's)
- LR per model (paper Table 8 -> small sweep on validation subset):
    - LLaMA3-8B: paper 1e-4, sweep [5e-5, 1e-4, 2e-4]
    - LLaMA2-7B: paper 2e-4, sweep [1e-4, 2e-4, 3e-4]
    - LLaMA-7B : paper 2e-4
- max_seq_len: 4096 (current config)

--- eval ---
- AIME answers are integers 0-999
- extract via \boxed{...}  (code/data.py::extract_boxed_answer)
- two settings: greedy + n-sample majority (n=16 or 64)
- pass@k for completeness
- SAME eval recipe across B0/B1/B2 + our scratch DoRA -- no per-method tweaks
- write all results to results/tables/<run_id>.json for the report

--- experiments ---
E1 - main (headline):
    scratch DoRA(r=16) vs peft LoRA(r=16) on AIME 2024 + 2025
    same train corpus, same LR, same epochs -> direct DoRA-vs-LoRA comparison
E2 - sanity:
    scratch DoRA(r=16) vs peft DoRA(r=16) at identical config
    must be within run-to-run noise -- validates our impl is faithful
E3 - zero-shot floor:
    pretrained Llama on AIME with no fine-tuning
    establishes the unadapted baseline (B0)
E4 - rank sweep (paper Fig 5):
    scratch DoRA + peft LoRA at r in {4, 8, 16, 32, 64}, alpha = 2r
    paper claim: DoRA's lead WIDENS at low rank (LoRA collapses at r<=8 on commonsense)
    -> does the same hold for math reasoning?
E5 - QDoRA (paper sec 6.1):
    scratch DoRA on 4-bit (NF4) base via bitsandbytes
    motivated by T4 Colab (16GB). paper: QDoRA > QLoRA by 0.19/0.23 on Orca-Math.
    open question: implement 4-bit dequant in our DoRALinear, or use peft's QDoRA wrapper?
