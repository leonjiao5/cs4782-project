\section{Introduction}
% ─────────────────────────────────────────────────────────────────

Large language model fine-tuning is essential for specializing pre-trained models to new domains, but full fine-tuning updates billions of parameters at prohibitive cost. Parameter-Efficient Fine-Tuning (PEFT) methods address this by training only a small number of adapter parameters while keeping the base model frozen.

\textbf{LoRA} \cite{hu2022lora} is the most widely used PEFT approach: it adds a low-rank update $\Delta W = BA$ ($B \in \mathbb{R}^{d \times r}$, $A \in \mathbb{R}^{r \times k}$, $r \ll \min(d,k)$) to a frozen weight matrix $W_0$. While effective, LoRA still shows a notable accuracy gap compared to full fine-tuning.

\textbf{DoRA} \cite{liu2024dora} closes this gap by decomposing $W_0$ into magnitude and direction, then applying LoRA only to the direction. The key insight from the paper is that full fine-tuning tends to make \emph{large magnitude changes and small direction changes}, while LoRA can only make direction changes. By explicitly separating the two components, DoRA more closely mimics full fine-tuning behavior with the same parameter budget.

We re-implement DoRA from scratch in PyTorch and evaluate it on \emph{competition mathematics}: AMC12 2024/2025 and the MATH benchmark --- a new domain not studied in the original paper. We compare our implementation against PEFT-LoRA and PEFT-DoRA baselines, and introduce a three-mode evaluation protocol that reveals a novel ``knows but can't say'' phenomenon in DoRA.

% ─────────────────────────────────────────────────────────────────
\section{Chosen Result}
% ─────────────────────────────────────────────────────────────────

We aim to reproduce the primary result of the DoRA paper: \textbf{DoRA consistently outperforms LoRA at the same rank/parameter budget} (paper Table 2, across NLP tasks including math and commonsense reasoning). A secondary claim is that DoRA at half rank ($r = 8$) matches full-rank LoRA ($r = 16$), demonstrating superior parameter efficiency.

We apply this to AMC12 2024/2025 (50 multiple-choice problems each, answers A--E) and the MATH benchmark (723 free-response problems across 5 difficulty levels). Our novel contribution is a three-scoring-mode evaluation protocol that decouples \emph{knowledge} from \emph{generation quality}, enabling finer-grained analysis than simple accuracy.

\section{Methodology}
% ─────────────────────────────────────────────────────────────────

\subsection{DoRA Algorithm}

DoRA adapts a frozen weight $W_0 \in \mathbb{R}^{d \times k}$ as:
\begin{equation}
  W' = \mathbf{m} \cdot \frac{W_0 + s \cdot BA}{\|W_0 + s \cdot BA\|_c}
  \label{eq:dora}
\end{equation}
where $B \in \mathbb{R}^{d \times r}$, $A \in \mathbb{R}^{r \times k}$ are low-rank LoRA factors initialized with $B = 0$ and $A$ Kaiming uniform; $s = \alpha / r$ is a scaling factor; $\mathbf{m} \in \mathbb{R}^{1 \times k}$ is a trainable per-column magnitude vector initialized to $\|W_0\|_c$; and $\|\cdot\|_c$ denotes the column-wise $\ell_2$ norm. Only $\mathbf{m}$, $B$, and $A$ are trained; $W_0$ is frozen.

\textbf{Critical implementation detail --- the detach trick} (Eq.~11 in \cite{liu2024dora}): the denominator $\|W_0 + s \cdot BA\|_c$ is \emph{detached} before backpropagation. Without detaching, gradients flow through the norm, increasing memory by 24.4\% on LLaMA-7B with only $\sim$0.2pp accuracy cost. Our implementation:
\begin{verbatim}
col_norm = weight_eff.norm(
    p=2, dim=0, keepdim=True
).detach().clamp_min(1e-12)
W_prime = (m / col_norm) * weight_eff
\end{verbatim}

\textbf{Merge/unmerge for zero inference cost:} after training, we call \texttt{merge()}, which fuses $W'$ back into the base layer weight and discards LoRA parameters, yielding a standard \texttt{nn.Linear} at inference with no latency overhead.

\subsection{Implementation}

We implement \texttt{DoRALinear}, a \texttt{nn.Module} that wraps a frozen \texttt{nn.Linear} layer and replaces its forward pass. The helper \texttt{apply\_dora\_to\_module()} recursively traverses a model and replaces target layers by name. This approach is framework-agnostic and does not depend on PEFT's internal structure.

We compare three methods:
\begin{enumerate}[leftmargin=*, noitemsep, topsep=2pt]
  \item \textbf{PEFT-LoRA}: HuggingFace PEFT \texttt{LoraConfig(use\_dora=False)}
  \item \textbf{DoRA (PEFT)}: HuggingFace PEFT \texttt{LoraConfig(use\_dora=True)} --- reference implementation
  \item \textbf{Custom DoRA}: our from-scratch PyTorch implementation
\end{enumerate}

\textbf{Base model:} Qwen2.5-3B (base and instruct variants). Target modules: \texttt{q\_proj, k\_proj, v\_proj, up\_proj, down\_proj} --- 5 of 7 attention/FFN projections. Total trainable parameters: $\sim$0.71\% of the model. Full implementation in Appendix~\ref{app:dora}.

\subsection{Datasets}

\textbf{Training tiers} --- all rows share the schema \texttt{\{problem, solution, answer, source\}}:
\begin{itemize}[leftmargin=*, noitemsep, topsep=2pt]
  \item \texttt{train\_light}: MATH-lighteval \cite{matheval} ($\sim$7,498 rows; pure math: algebra, geometry, number theory, counting, precalc). Each row contains a full step-by-step solution ending in \texttt{\textbackslash boxed\{answer\}}.
  \item \texttt{train\_heavy}: \texttt{train\_light} + NuminaMath-CoT \cite{numina} subset ($\sim$27,500 total). NuminaMath adds physics, proof-style, and olympiad problems from a wider range of sources.
  \item \texttt{train\_s1k}: 1,000-row subset of \texttt{train\_light}.
\end{itemize}

\subsection{Training Setup}

We use TRL's \texttt{SFTTrainer} for supervised fine-tuning on (problem, solution) pairs. Hyperparameters: rank $r = 8$, $\alpha = 32$, dropout $= 0.05$, learning rate $1 \times 10^{-4}$, cosine LR schedule, 3\% warmup, effective batch size 16 (per-device 1, gradient accumulation 16), 1--3 epochs.

\subsection{Novel Evaluation Protocol}

Standard greedy accuracy conflates knowledge with generation quality. We introduce three scoring modes:

\begin{enumerate}[leftmargin=*, noitemsep, topsep=2pt]
  \item \textbf{Logit scoring}: append ``\texttt{\textbackslash nThe answer is (}'' to the prompt $\to$ single forward pass $\to$ read logits for tokens A/B/C/D/E $\to$ argmax. A pure \emph{knowledge probe}: does the model assign highest probability to the correct answer token?
  \item \textbf{Two-pass scoring}: generate a full reasoning chain, then append the suffix and read answer logits. Tests whether reasoning improves answer selection.
  \item \textbf{Greedy scoring}: generate a full response (up to 2048 tokens) and parse the $\backslash$\texttt{boxed\{\}} answer. Standard \emph{generation quality} metric.
\end{enumerate}

This protocol lets us ask: does a model \emph{know} the answer (logit) but \emph{fail to say it} (greedy)? As we show below, this distinction is the central finding of our work. Figure~\ref{fig:explainer} illustrates logit scoring visually. Implementation is in Appendix~\ref{app:eval}.

\subsection{Key Findings}

\textbf{Finding 1 --- ``DoRA knows but can't say'' (main finding).}
The logit--greedy gap is consistently and significantly larger for DoRA than LoRA at the same training configuration. DoRA light shows $\Delta = +26$pp vs.\ LoRA light's $\Delta = +20$pp. This means DoRA assigns high probability to the correct answer token internally, but fails to produce it through generation. We hypothesize that DoRA's magnitude--direction decomposition reorganizes how the model stores knowledge in a way that makes it harder to verbalize, even when the representation is correct.

\textbf{Finding 2 --- More training data closes, and eventually reverses, the logit--greedy gap.}
Both LoRA and DoRA benefit from more data, but in different ways. LoRA heavy (3.7$\times$ more data) gradually closes its gap: logit $\approx$ greedy ($34\% \approx 36\%$, $\Delta = -2$pp). DoRA heavy shows a more dramatic shift: greedy ($46\%$) \emph{exceeds} logit ($32\%$), $\Delta = -14$pp --- the gap fully reverses. DoRA heavy achieves the highest greedy accuracy of any fine-tuned model. Interestingly, while both methods move in the same direction (more data $\to$ greedy improves relative to logit), DoRA's swing is much larger, going from $\Delta = +26$pp (light) to $\Delta = -14$pp (heavy). This suggests DoRA particularly benefits from additional training signal to learn fluent generation, while LoRA converges more smoothly.

\textbf{Finding 3 --- Knowledge is rank-invariant for DoRA.}
Across r2, r4, and r8, DoRA logit accuracy stays near 42--44\%. Knowledge representation is captured regardless of rank. However, DoRA greedy accuracy remains low at all ranks (12--16\%), while LoRA greedy improves more readily with rank. This suggests the logit--greedy gap is a property of DoRA's adaptation mechanism, not its capacity.

\textbf{Finding 4 --- MATH regression is worse for DoRA.}
Fine-tuning on AMC-style data hurts general math reasoning. DoRA light falls 13pp below baseline (30.7\% vs.\ 43.8\%), worse than LoRA heavy (35.0\%). Notably, LoRA s1k (1K-example instruct fine-tune) nearly preserves MATH (44.8\%), suggesting small targeted datasets cause less catastrophic interference with general knowledge. DoRA's magnitude decomposition may cause more aggressive weight modification, amplifying this effect.

\textbf{Finding 5 --- Partial replication of the paper.}
The original DoRA paper demonstrates DoRA $>$ LoRA on 7B models across NLP tasks. At the logit (knowledge) level, we reproduce this: DoRA r4 matches the best LoRA logit result (both 44\%). However, greedy accuracy does not confirm DoRA $>$ LoRA at 3B scale. Likely causes: (a) model scale (3B vs.\ 7B in paper), (b) math is more domain-specific than commonsense, (c) our training data is smaller. The paper's claims may require larger models to fully manifest.

\section{Reflections}
% ─────────────────────────────────────────────────────────────────

\textbf{Logit scoring as a knowledge probe.}
The three-mode evaluation protocol was our most important methodological contribution. Standard greedy evaluation would have shown only that DoRA underperforms LoRA on generation. Logit scoring revealed that DoRA actually \emph{learns the right answer} --- it just cannot verbalize it. Two-pass scoring (generate reasoning, then read logits) provides an intermediate point: DoRA two-pass consistently falls between logit and greedy, suggesting reasoning chains partially help but do not fully close the generation gap. This insight opens a new research question: can logit-guided decoding recover DoRA's latent knowledge?

\textbf{Scale limitations.}
The original paper evaluates on 7B--13B models with large, high-quality datasets (e.g., MetaMathQA, commonsense datasets). At 3B with $\sim$7.5K training problems, our setup is a harder regime where PEFT methods struggle more, and the gains from DoRA over LoRA may not yet be visible in generation. Our results suggest 3B is below the threshold where DoRA's advantages over LoRA clearly manifest in generation quality.

\textbf{Future directions.}
\begin{enumerate}[leftmargin=*, noitemsep, topsep=2pt]
  \item \textbf{Larger models}: 7B+ to match the paper's regime, where DoRA's generation advantage may materialize.
  \item \textbf{Harder benchmarks}: AIME 2024/2025 (30 free-response problems, much harder than AMC12).
  \item \textbf{Decoding interventions}: given DoRA ``knows but can't say,'' can logit-guided or constrained decoding recover the latent knowledge?
  \item \textbf{Mechanistic understanding}: why does DoRA's magnitude decomposition create a larger logit--greedy gap? Probing the magnitude vectors ($\mathbf{m}$) may reveal which layers and features drive this behavior.
\end{enumerate}

\section{DoRA Implementation (\texttt{code/dora\_layers.py})}
\label{app:dora}

The full \texttt{DoRALinear} layer and model injection helper. The key equations are in \texttt{\_compute\_scaled\_weight}: the LoRA update is added to the frozen base weight, the column-wise norm is computed and detached, and the magnitude vector $\mathbf{m}$ rescales the result.

\begin{lstlisting}
class DoRALinear(nn.Module):
    def __init__(self, base: nn.Linear, rank: int,
                 alpha: float, dropout: float):
        super().__init__()
        self.base = base          # frozen pretrained weight W_0
        self.scaling = alpha / rank  # LoRA scaling factor s = alpha/r
        in_f, out_f = base.in_features, base.out_features
        dev, dtype = base.weight.device, base.weight.dtype

        # Low-rank factors: B (d x r) init=zeros, A (r x k) Kaiming
        # At init, BA=0 so W' = W_0 (no change until training starts)
        self.lora_A = nn.Parameter(
            torch.zeros((rank, in_f), device=dev, dtype=dtype))
        self.lora_B = nn.Parameter(
            torch.zeros((out_f, rank), device=dev, dtype=dtype))
        nn.init.kaiming_uniform_(self.lora_A, a=0)

        with torch.no_grad():
            # m initialized to ||W_0||_c so W' = W_0 at step 0
            self.m = nn.Parameter(
                base.weight.norm(p=2, dim=0, keepdim=True).clone())

        # Freeze base weight: only m, lora_A, lora_B are trainable
        for p in self.base.parameters():
            p.requires_grad = False
        self.dropout = nn.Dropout(p=dropout)

    def _compute_scaled_weight(self) -> torch.Tensor:
        # Step 1: compute the effective weight W_0 + s*BA (Eq. 5 numerator)
        lora_update = (self.lora_B @ self.lora_A) * self.scaling
        weight_eff  = self.base.weight + lora_update

        # Step 2: column-wise L2 norm, detached (Eq. 11 detach trick).
        # Detaching stops gradients flowing through the norm denominator,
        # saving ~24% memory at ~0.2pp accuracy cost (per paper ablation).
        col_norm = weight_eff.norm(
            p=2, dim=0, keepdim=True
        ).detach().clamp_min(1e-12)  # clamp avoids division by zero

        # Step 3: rescale by learned magnitude m --- Eq. 5
        return (self.m / col_norm) * weight_eff

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        if self.merged:
            # After merge(), W' is already baked into base.weight
            return self.base(x)
        # Apply dropout to input, then matmul with DoRA-scaled weight
        return F.linear(self.dropout(x),
                        self._compute_scaled_weight(),
                        self.base.bias)

    def merge(self) -> None:
        """Bake W' into base.weight so inference is a plain Linear."""
        self._cached_base_weight = self.base.weight.detach().clone()
        with torch.no_grad():
            self.base.weight.copy_(self._compute_scaled_weight())
        self.merged = True
\end{lstlisting}

\begin{lstlisting}
def apply_dora_to_module(model, target_names,
                         rank, alpha, dropout=0.05):
    """Walk the model tree; swap matching Linear -> DoRALinear."""
    target_set = set(target_names)  # e.g. {"q_proj","v_proj",...}
    def recurse(module, prefix=""):
        for name, child in module.named_children():
            qname = f"{prefix}.{name}" if prefix else name
            if (isinstance(child, nn.Linear)
                    and (name in target_set
                         or qname in target_set)):
                # In-place replacement: parent.child = DoRALinear(child)
                setattr(module, name,
                        DoRALinear(child, rank, alpha, dropout))
            else:
                recurse(child, qname)  # recurse into sub-modules
    recurse(model)
    return model
\end{lstlisting}

\section{Evaluation Scoring Modes (\texttt{code/eval.py})}
\label{app:eval}

The three scoring functions used to decouple knowledge from generation quality.

\begin{lstlisting}
MC_LETTERS = ["A", "B", "C", "D", "E"]
# Appending this suffix steers the model to predict a letter next
SUFFIX = "\nThe answer is ("

def _letter_logits(model, tokenizer, text):
    """Core of logit scoring: one forward pass, read next-token logits."""
    # Tokenize prompt + suffix; model sees the full context
    inputs = tokenizer(text + SUFFIX,
                       return_tensors="pt").to(model.device)
    # Map each letter to its token ID (single-token for A-E)
    letter_ids = {l: tokenizer.encode(
        l, add_special_tokens=False)[0] for l in MC_LETTERS}
    with torch.no_grad():
        # logits[0, -1, :] = distribution over vocab at the *last* position
        last_logits = model(**inputs).logits[0, -1, :]
    # Extract raw logit (not probability) for each answer letter
    scores = {l: last_logits[tid].item()
              for l, tid in letter_ids.items()}
    # argmax over A-E gives the model's most likely next token
    return max(scores, key=scores.__getitem__), scores

def score_mc_logit(model, tokenizer, prompt):
    """Logit scoring: no generation, pure knowledge probe."""
    return _letter_logits(model, tokenizer, prompt)

def score_mc_twopass(model, tokenizer, prompt,
                     max_new_tokens=4096):
    """Two-pass: let model reason first, then read answer logits."""
    inputs = tokenizer(prompt, return_tensors="pt").to(model.device)
    prompt_len = inputs["input_ids"].shape[1]
    with torch.no_grad():
        # Pass 1: generate a full reasoning chain (greedy)
        out = model.generate(**inputs, do_sample=False,
                             max_new_tokens=max_new_tokens)
    # Decode only the newly generated tokens (skip the prompt)
    reasoning = tokenizer.decode(out[0][prompt_len:],
                                 skip_special_tokens=True)
    # Pass 2: append reasoning to prompt, re-read answer logits
    return _letter_logits(model, tokenizer, prompt + reasoning)

def generate_responses(model, tokenizer, prompt, n,
                       temperature, max_new_tokens=4096):
    """Greedy scoring: generate full response, caller parses \\boxed{}."""
    inputs = tokenizer(prompt, return_tensors="pt").to(model.device)
    prompt_len = inputs["input_ids"].shape[1]
    # Greedy (temp=0) or sampled (temp>0, for majority-vote pass@k)
    gen_kwargs = (dict(do_sample=False, num_return_sequences=1)
                  if n == 1 or temperature == 0.0
                  else dict(do_sample=True, temperature=temperature,
                            num_return_sequences=n))
    with torch.no_grad():
        out = model.generate(**inputs, **gen_kwargs,
                             max_new_tokens=max_new_tokens)
    # Slice off the prompt tokens; return only generated text
    return [tokenizer.decode(seq[prompt_len:],
                             skip_special_tokens=True)
            for seq in out]
